# Domain 레이어 규칙

## 역할

- Domain은 회원, 그룹, 일정처럼 업무에서 다루는 대상의 데이터와 의미를 표현한다.
- Domain은 업무 데이터가 가질 수 있는 상태와 상태를 바꾸는 규칙을 표현한다.
- Domain은 계산 결과와 상태 변경 후에도 반드시 지켜야 하는 조건을 보장한다.

## 구현 규칙

- Domain 클래스와 enum은 업무에서 사용하는 이름으로 작성한다.
- Domain 객체의 상태를 바꾸는 규칙은 그 객체 또는 같은 Domain 레이어의 함수에 둔다.
- Domain 객체는 생성할 때와 상태를 바꿀 때 반드시 지켜야 하는 조건을 확인한다.
- 같은 입력값에서 항상 같은 결과가 나오는 계산은 Domain에 둔다.
- Domain enum에는 업무상 허용된 값만 둔다.
- 문자열이나 숫자를 Domain 값으로 바꾸는 함수는 허용하지 않는 값을 만났을 때의 동작을 명확히 정한다.
- 데이터베이스 테이블에 저장되는 Domain 객체는 테이블과 열을 연결하기 위한 JPA 매핑 어노테이션을 가질 수 있다.

## 금지 규칙

- Domain은 Controller, Service, Repository를 참조하지 않는다.
- Domain은 HTTP 요청·응답, Servlet, Discord 이벤트, 파일 업로드 요청 타입을 사용하지 않는다.
- Domain은 외부 API와 메시지 시스템을 호출하지 않는다.
- Domain은 데이터베이스에서 데이터를 조회하거나 저장하지 않는다.
- Domain은 환경 변수, 설정 파일, 현재 로그인한 사용자의 정보를 직접 읽지 않는다.
- Domain은 HTTP 상태 코드, 응답 DTO, 화면 표시 형식을 결정하지 않는다.
- Domain은 현재 시간, 난수, 네트워크 결과처럼 호출할 때마다 값이 달라질 수 있는 값을 직접 만들거나 읽지 않는다.

## 의존성 규칙

- Domain은 다른 Domain 타입, Kotlin 표준 라이브러리, JPA 매핑 어노테이션에만 의존한다.
- Domain의 업무 규칙은 HTTP 요청 처리 순서와 데이터베이스 호출 순서에 의존하지 않는다.
- Domain의 공개 메서드는 HTTP, SQL, JSON 같은 기술 용어를 업무 개념 대신 사용하지 않는다.
